# aksaraLLM Governance

## Overview

aksaraLLM uses a **meritocratic open governance model** inspired by the Apache Software Foundation and EleutherAI. This document describes how decisions are made and how the project is structured.

## Principles

1. **Transparency** — All decisions are made in public (GitHub, Discord)
2. **Meritocracy** — Influence is earned through contributions
3. **Consensus** — We seek consensus; voting is a last resort
4. **Inclusivity** — Everyone's voice matters regardless of background

## Roles & Responsibilities

### 🌱 Contributor
**Anyone who contributes to the project in any form.**

- Submit issues, PRs, documentation, data, research
- Participate in discussions
- No special access required — just fork and contribute

### 🔧 Committer
**Regular contributors who have earned write access.**

Criteria to become a Committer:
- At least 5 merged PRs with meaningful contributions
- Consistent participation over 1+ months
- Nominated by an existing Committer or Maintainer
- Approved by majority vote of Core Team

Responsibilities:
- Review and merge PRs in their area
- Triage issues
- Mentor new contributors

### 🛡️ Maintainer
**Leads a specific repository or area.**

Criteria:
- Committer for 2+ months
- Deep expertise in their area
- Nominated by Core Team

Responsibilities:
- Set technical direction for their repo
- Final say on merges in their area (with consensus)
- Represent their area in Core Team discussions

## Decision Making

### Day-to-Day Decisions
- Made by relevant Maintainer or Committer
- Seek input from others when impact is broad
- Document decisions in relevant GitHub issues

### Significant Decisions (via RFC)
Used for:
- Architecture changes
- Data composition changes
- Training methodology changes
- New repository creation
- Governance changes

Process:
1. Author opens an RFC issue using the template
2. Minimum 1-week discussion period
3. Core Team reviews and decides:
   - ✅ **Accept** — move to implementation
   - 🔄 **Revise** — author revises based on feedback
   - ❌ **Reject** — with documented reasoning
4. Decision is logged in the RFC issue

### Voting
When consensus cannot be reached:
- Core Team members each get 1 vote
- Simple majority wins
- Project Lead breaks ties
- Votes are public and recorded

## Conflict Resolution

1. **Direct discussion** between parties
2. **Mediation** by a Maintainer
3. **Core Team review** if unresolved
4. **Project Lead decision** as last resort
